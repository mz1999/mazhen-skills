# JDK 源码级技术审计与原理讲解

## 这个 skill 要解决什么

用户以**第三方 JDK 专家**身份面向客户交付。回答必须同时满足：

- **严谨准确**：交付结论的权威来源只有源码。自身理解、网上文章在通用层面大多正确，但在特定发行版（如毕昇 21）上可能有偏差——这种"似是而非"比完全错误更难察觉。因此每个结论必须有源码证据（`文件:行号`），不能用"一般来说""根据我的知识"或网上文章替代。一条总规则：**你的预判和查到的资料都只是线索，结论只认源码**。预判帮你列"该看哪些源码"的清单，联网搜索帮你定位源码、查设计意图的历史记录（JEP/commit）——但它们都不出现在交付结论里作依据。
- **深入浅出**：先让自己完全搞懂，才能给非专家讲清楚。

两个要求靠一条主线统一：**先想透"为什么"（根本目的 → 必然设计），再讲清"为什么"（让非专家跟上）。** 前者是分析，后者是表达，但回答的是同一条"为什么"链。

## 分析阶段

分析的目标：把"为什么"想透，并用源码为每个结论盖章。

> **关于联网搜索**：搜索是手段不是权威。三个点上有用——查 JEP/官方文档理解设计意图（Step 1/3）、定位源码（Step 2）、核查预判是否过时（Step 2）。但搜到的任何东西，用到结论里前必须回到源码验证。

### Step 1: 立主线（先于源码）

碰任何源码之前，先回答三件事——这是整篇的骨架：

1. **这个系统/机制为什么存在？** 要完成什么根本任务？
2. **它遇到什么现实约束/矛盾？**（如"要回收内存，但不能停顿太久"）
3. **由此能推导出哪些必然的设计？** 写下根本目的 → 必然设计的推导链。

推导用**纵向瀑布**：根本目的 → 遇到什么现实约束（矛盾）→ 约束逼出什么必然选择 → 这个选择引入什么新问题 → 新问题逼出下一个机制……直到核心机制都被"长"出来。

> 示例（ZGC 从一根线长出来）：GC 要回收内存 → 但不能只释放垃圾（会碎片化）→ 必须整理 → 这拆成两件 O(堆) 的重活：**找活的（标记）** 和 **挪活的（搬运）** → 太重不能 STW，只能**并发** → 并发就要协调冲突 → 于是**翻转、屏障、转发表、lazy remap** 全部被逼出来。

讲单点机制时，主线退化为"这个机制解决什么矛盾、它出现之前会怎样、它出现之后带来什么"。

然后做两件事，为后续源码核查铺路：

1. **列出"预期会有的相关机制 / 该担心的边界"清单**——凭你对这类系统的理解列出（如"G1 调整新生代大小，预期会有最小/最大边界参数、和暂停目标的联动"）。这张清单不是结论，是后续去源码里逐个核实的索引。
2. **查 JEP / 官方文档**：对关键设计决策（如"ZGC 为什么用染色指针"），联网查设计意图，建立准确的心智模型，避免凭源码现状事后反推。

### Step 2: 源码定位（验事实，并做足充分性）

**先确认源码根路径**：默认当前工作目录就是 OpenJDK / 毕昇 JDK 源码仓库，源码引用按仓库相对路径给出（如 `src/hotspot/share/gc/z/ZAddress.cpp:130`）。若用户指定了其他路径，以用户指定为准。开始检索前先确认根路径下存在 `src/hotspot`——若不存在，向用户确认路径，别在错误的目录里空搜。

确认路径后，为主线上的每个结论、每个机制去找源码证据。

**源码充分性是这一步的核心要求——直击"只看局部"这个最大失败模式：**

- 按 Step 1 清单逐项去源码里找，能验证的验证，找不到的要警觉。
- 不只看一个函数：追踪调用链，读调用方和被调用方。
- 不只看一条分支：读默认值、读所有条件分支、读边界情况——这些往往是真正决定行为的地方。
- 读关键片段的完整函数体和上下文，不只看一行。

找不到某个预期机制，先别下"不存在"的结论——它可能在别的文件、别的类、或用别的名字实现。**用搜索（类名、关键字、调用关系）在源码库里定位它**。

**预判-源码冲突时**：Step 1 清单里预期的机制在源码里怎么都找不到，搜一下 JEP/官方文档，确认它在该版本是否还存在、或换了实现方式——可能是你的预判过时，也可能是你漏看了，两种都查清楚再下结论。

**讲"为什么这样设计"时**：源码只呈现"现在的结果"，答不了"它为什么这么做"。查 JEP/commit 吃透设计意图，但毕昇是否照实现仍以源码为准。

### Step 3: 差异识别

如果是毕昇 JDK，读 `references/bisheng-diff.md`，按其方法判断是 OpenJDK 通用逻辑还是华为新增/修改。

**源码解释不了"为什么从 A 改成 B"时**：查 commit message、JEP、官方文档，理解设计变更的来龙去脉。但记住：这些是帮你看懂决策过程，毕昇是否照此实现仍以源码为准。

### Step 4: 提炼与组织

不要读完源码就直接输出。先按主线把材料组织成五要素：

1. **靶心**：一句话概括根本矛盾
2. **主线**：纵向推导链，或横向因果链（机制间联动），按场景选
3. **源码证据**：每个关键论点绑定源码位置
4. **联动分析**：改了 A，B 和 C 会受什么影响
5. **实践判断**：什么场景要关注，调大/调小的后果

**横向因果链示例**：因为 Young GC 不碰老年代 → 所以不知道哪些老年代 Region 垃圾多 → 所以需要 Concurrent Mark 建"垃圾率地图" → 因为地图建好 → 所以 Mixed GC 按图分批回收。

**提炼示例（用户问 G1NewSizePercent 和 G1MaxNewSizePercent）：** 不要只答"一个最小占比一个最大占比"。按主线讲：为什么需要这两个参数（平衡 GC 频率和耗时）→ 堆布局前提（Region-based）→ 联动（`G1HeapRegionSize`、`MaxGCPauseMillis`、`G1MixedGCCountTarget`）→ 默认值的来由（5% 和 60% 基于实验数据，非拍脑袋）→ 什么场景调。

## 表达阶段

分析做完，讲给别人听还要降低门槛。这一阶段遵循 `references/expression.md`：源码引用如何绑定论点、用因果链串知识点、形象化类比与三层递进、复杂内容配图配表。

**表达阶段的铁律：输出第一段应是"根本目的/为什么"，不是机制名称或源码。** 读者第一眼看到"GC 为什么要回收内存"，不是 `zAddress.cpp:130`。源码是证据不是叙事，服务于"为什么"这条主线，不反过来主导。

## 落稿后

落稿后读 `references/checklist.md` 过自检清单（主线、证据、表达、按场景的完整性维度）。
