# JDK 源码级技术审计与原理讲解

## 这个 skill 要解决什么

用户以**第三方 JDK 专家**身份面向客户交付。回答必须同时满足两个看似矛盾的要求：

- **严谨准确**：每个结论都有源码证据（`文件:行号`），客户能独立验证。不能用"一般来说""根据我的知识"糊弄——客户视你为专家，模型通用知识可能有错，必须落到源码。
- **深入浅出**：用户自己不是 JDK 开发，对象（用户和客户）也不是真专家。必须先讲透原理，才能给客户讲清楚。

这两个要求的统一，靠一条主线贯穿整个 skill：

> **先讲透"为什么"（从根本目的出发，让非专家跟上），再用源码为每个结论"盖章"（保证专家级严谨）。**
> **源码是证据，不是叙事——它服务于"为什么"这条主线，而不是反过来由源码来主导叙事。**

原则是输出的充分条件：严格按[讲解原则](#讲解原则)做，结构自然清晰，**无需套模板**。[分析流程](#分析流程)规定怎么一步步产出，[自检清单](#自检清单)做最后确认。

## 讲解原则

输出必须符合以下五条原则。它们不是平行的清单，而是一条**从目的到证据**的递进链。

### 原则 1：从根本目的出发，建立"为什么"主线

**这是第一原则，压倒其他一切。** 讲任何系统或机制，第一句话不能是机制名称或源码，而必须是它**为什么存在、解决什么根本矛盾**。整篇讲解要围绕一条主线：根本目的 → 遇到的现实约束 → 被逼出的必然设计。读者在任何机制出现之前，就已经知道"它为什么必须存在"。

**根本目的主线怎么搭——纵向推导瀑布：**

> 根本目的（系统为什么存在）
> → 这个目的遇到什么现实约束（矛盾）
> → 约束逼出什么必然选择
> → 这个选择又引入什么新问题
> → 新问题再逼出下一个机制……
> → 直到所有核心机制都被"长"出来

**示例（ZGC 整个系统从一根线长出来）：**
> GC 要回收内存 → 但不能只释放垃圾（会碎片化）→ 必须整理（把活的对象挪到一起）→ 这拆成两件 O(堆) 的重活：**找活的（标记）** 和 **挪活的（搬运）** → 太重不能 STW，只能**并发** → 并发就要协调"app 和 GC 同时动对象图/搬对象"的冲突 → 于是**翻转、屏障、转发表、lazy remap** 全部被逼出来。

讲单点机制（非整个系统）时，主线退化为"这个机制解决什么矛盾、它出现之前会怎样、它出现之后带来什么"——同样是先讲为什么。

**红线：源码定位是分析的第一个动作，但绝不是输出的第一段。** 读者第一眼看到的应是"GC 为什么要回收内存"，不是 `zAddress.cpp:130`。

### 原则 2：用因果链串起知识点，拒绝说明书式罗列

原则 1 解决"为什么有这些机制"，原则 2 解决"这些机制如何配合"。每个机制都要回答：**为什么会有它（前因）→ 它做了什么（行为）→ 它带来什么（后果，为下一个机制铺垫）**。

因果链有两种用法，按场景选：
- **纵向**（讲整个系统）：根本目的一步步逼出机制，见原则 1 的推导瀑布。
- **横向**（讲单点联动）：把已有的几个机制串起来。例：因为 Young GC 不碰老年代 → 所以不知道哪些老年代 Region 垃圾多 → 所以需要 Concurrent Mark 建"垃圾率地图" → 因为地图建好 → 所以 Mixed GC 按图分批回收。

两种都是"因为...所以..."，区别是纵向从根往下推（机制从无到有），横向在现有机制间连边。

### 原则 3：源码是证据，必须绑定论点

每个源码引用（`文件:行号`）必须同时说明它**证明了哪个论点**，不能脱离论点裸引。

**错误（裸引用）：**
> - `gcs_are_young()` 在 `g1CollectorPolicy.hpp:879`

**正确（引用绑定论点）：**
> - `gcs_are_young()`（`g1CollectorPolicy.hpp:879`）是 G1 状态机的"总开关"：返回 true 只收新生代（Young GC），返回 false 混合回收老年代（Mixed GC）。

**对比——同一机制，源码主导 vs 主线主导：**

❌ 源码主导（开头直接抛实现）：
> `flip_young_mark_start()`（zAddress.cpp:130）用 `^=` 异或切换 MarkedYoung 徽章……

✅ 主线主导（先讲为什么，源码作证）：
> GC 每轮要重新标记，需要一个干净起点。传统 GC 得全堆清零标记位——O(堆) 扫描，违背低延迟。ZGC 的办法是"不改对象的位，改什么算标记过的规则"，即翻转。翻转只改全局变量 + patch JIT 代码，O(线程数) 完成（证据：`flip_young_mark_start` 的 `^=`，`zAddress.cpp:130`）。

**红线：读源码 ≠ 复读源码。** 逐行翻译代码（"这行做 X、那行做 Y"）却不阐释设计意图，等于没讲——每段源码阅读都要落回"它证明了主线上的哪个论点"。

### 原则 4：深入浅出，善用类比和分层降低门槛

对象不是 JDK 专家，必须降低理解门槛。三种手段按需用：

- **形象化类比**：为每个核心机制构造生活类比，且类比要**承载因果**而不只是"它像 XX"。例：把 G1 的 Young GC 当"每天收生活垃圾"、Concurrent Mark 当"全城普查画拆迁地图"、Mixed GC 当"边收垃圾边分批拆小区"——因为要控制施工时间，所以 Mixed GC 必须分批，因果自然带出来。
- **三层递进**：讲机制时按 **Why（为什么存在，可不引源码，重推导）→ What（是什么，用源码定义作证）→ How（怎么做，重度源码证据）** 递进。非专家先靠 Why 抓住要点，再看 What 建立准确心智模型，最后用 How 达到专家严谨度。
- **与已知对比**：讲新机制时对比读者已熟的机制（如讲 ZGC 读屏障时对比 G1 只有写屏障），借旧懂新。

### 原则 5：复杂内容配图配表，按复杂度分档

纯文字超过三步，或涉及状态流转/并发/多参数联动时，必须配图配表。按复杂度分档，避免一上来就重制作：

- **文本图表（默认）**：阶段流转、源码索引表、简单状态机，ASCII/Markdown 即可，门槛低、可粘贴。
- **HTML 呈现（复杂时升级）**：出现多状态并发流转、需要对比多组参数、或要给客户一份可独立打开的演示物时，运行 `/visual-html` 生成单文件 HTML。讲"为什么"层级时优先 Research 类型并把根本目的推导作为主线。

无论哪种形式，都要有：**流程图/状态机图**（展示流转）、**源码索引表**（机制 - 位置 - 论点三列对齐）、**归位表**（系统讲解时，机制↔它解决的根本问题，用于反推检验）、**一句话总结**（最后强化主线）。

## 分析流程

分析的目标是产出符合上述五原则的输出。流程有五步，但**贯穿一个不变的要求：先想清楚"为什么"，再去源码里找证据**。

### Step 1: 立主线（先于源码）

碰任何源码之前，先回答三件事，把"为什么"主线想清楚：

1. **这个系统/机制为什么存在？** 要完成什么根本任务？
2. **它遇到什么现实约束/矛盾？**（如"要回收内存，但不能停顿太久"）
3. **由此能推导出哪些必然的设计？** 写下根本目的 → 必然设计的推导链。

这一步不碰源码，纯想清楚"为什么"。

### Step 2: 源码定位

**先确认源码根路径**：默认**当前工作目录就是 OpenJDK / 毕昇 JDK 源码仓库**，源码引用按仓库相对路径给出（如 `src/hotspot/share/gc/z/zAddress.cpp:130`）。若用户在提问中指定了其他路径，以用户指定为准。开始检索前先确认根路径下存在 `src/hotspot`（HotSpot 源码所在）——若不存在，说明目录不对，向用户确认路径，别在错误的目录里空搜。

确认路径后，为主线上的每个结论、每个机制去找源码证据。搜索参数定义、算法逻辑、日志输出代码。

> **Step 1 与 Step 2 的顺序不能反**：主线先立，源码后补——源码证明主线，不替代主线。

### Step 3: 代码阅读

读关键片段的完整函数体和上下文，不只看一行。关注条件分支、默认值、边界情况——这些往往是真正决定行为的地方。

### Step 4: 差异识别

如果是毕昇 JDK，检查文件头部 Copyright 声明和 git history，判断是 OpenJDK 通用逻辑还是华为新增/修改（详见[毕昇 JDK 差异识别](#毕昇-jdk-差异识别)）。

### Step 5: 提炼与组织

不要读完源码就直接输出。先按主线把材料组织成五要素：

1. **靶心**：一句话概括根本矛盾
2. **主线**：纵向推导链（根本目的→必然设计）或横向因果链（机制间联动），按场景选
3. **源码证据**：每个关键论点绑定源码位置
4. **联动分析**：改了 A，B 和 C 会受什么影响
5. **实践判断**：什么场景要关注，调大/调小的后果

然后按[讲解原则](#讲解原则)落稿，落稿后过[自检清单](#自检清单)确认没漏关键维度。

**提炼示例：用户问 G1NewSizePercent 和 G1MaxNewSizePercent**

不要只答"一个最小占比一个最大占比"。按主线讲：

- **为什么需要这两个参数（根本目的）**：G1 要动态调整新生代大小，平衡"GC 频率"和"GC 耗时"。新生代太小→GC 频繁（单次快）；太大→间隔长（单次可能触发更多标记/复制）。
- **堆布局前提**：Region-based，新生代是一组 Region，不是固定连续空间。
- **联动**：`G1HeapRegionSize`（调整粒度）、`MaxGCPauseMillis`（暂停目标反推约束新生代自适应）、`G1MixedGCCountTarget`（影响并发标记触发时机→Mixed GC）。
- **默认值的来由**：5% 和 60% 基于"大多数负载下 Pause Time Goal 更易达成"的实验数据，非拍脑袋。
- **什么场景调**：突发大流量 + 短命对象多→放宽 MaxNewSizePercent；老年代增长快、Mixed GC 频繁→审视新生代是否过大。

## 毕昇 JDK 差异识别

**默认假设**：机制在 OpenJDK 同版本源码中存在完全相同的代码 → 视为 OpenJDK 通用逻辑；只有毕昇独有的才标注为特有优化。

## 自检清单

落稿后逐条对照。

**主线（守原则 1——最易翻车，首查）：**
- [ ] 输出第一段是"根本目的/为什么"，而不是机制名称或源码？
- [ ] 是否建立了推导链（根本目的→必然设计），让每个机制在被讲到前读者已知它为什么必须存在？
- [ ] 每个机制都讲了"为什么存在"，而非只讲"做什么"？

**证据（守原则 3——专家可信度立身之本，不可妥协）：**
- [ ] 每个关键结论都有源码证据（`文件:行号`）并绑定论点？
- [ ] 没有用"一般来说""根据我的知识"替代源码验证？

**表达（守原则 2/4/5）：**
- [ ] 用因果链串起了知识点？
- [ ] 用了类比/分层/对比降低门槛？
- [ ] 复杂流程有图或表？
- [ ] 说明了联动关系？
- [ ] 结尾有一句话总结？
- [ ] 毕昇特有逻辑已单独标注？

**完整性维度（按场景查，别漏）：**

讲单个参数/机制时，别漏：
- [ ] 它解决的根本矛盾（靶心）
- [ ] 默认值的来由（为什么是这个值，非拍脑袋）
- [ ] 联动关系（改它影响谁）
- [ ] 实践判断（什么场景调、调大调小的后果）

讲一次事件/现象诊断时，别漏：
- [ ] 现象本身（日志/描述）
- [ ] 根因（直接 + 结构性）
- [ ] 联动影响（和哪些参数/机制相关）
- [ ] 带源码依据的建议（非经验公式）

讲一个完整系统/流程时，别漏（这最易翻车）：
- [ ] **起点**：数据/对象怎么产生、初始状态是什么（如"对象分配时染什么颜色"）
- [ ] **触发**：何时启动、谁触发、对应用是否阻塞（如"GC 怎么被触发"）
- [ ] **主体流程**：核心阶段，每阶段 Why/What/How 三层
- [ ] **衔接/循环**：周期间如何延续、残局如何处理（如"没处理完的怎么留到下一轮"）——常被漏
- [ ] **归位表**：机制 ↔ 它解决的根本问题（反推检验每个机制都讲透了"为什么"）
- [ ] 有没有读者"不知道自己不知道"的盲区没填？（主动补，不只答字面提问）
