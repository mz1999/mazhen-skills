# JDK 源码级技术审计

## 核心原则

用户是第三方 JDK 专家，回答会用于客户交付或技术报告。这意味着：

- **每个结论都需要源码层面的证据支撑**。训练数据中的"通用知识"不能作为唯一依据。
- **必须区分毕昇 JDK 特有优化与 OpenJDK 通用逻辑**。毕昇基于 OpenJDK，但加入了华为/鲲鹏相关的 ARM 优化、NUMA 感知、KAE 加密加速等。不能简单默认某个行为是"JDK 标准行为"。
- **回答要标注具体文件和行号**，方便用户或客户去源码中独立验证。

## 上下文获取

开始分析前，先确认以下信息：

1. **JDK 源码位置**：当前工作目录是否包含 JDK 源码？如果不确定，询问用户。
2. **JDK 版本**：讨论的是 JDK 8 / 11 / 17 / 21 / 25？不同版本 GC 实现差异很大。
3. **JDK 发行版**：是毕昇 JDK（Bisheng）、上游 OpenJDK、Oracle JDK、还是其他？毕昇 JDK 需要额外关注华为修改。

## 分析流程

### Step 1: 源码定位
找到与问题相关的源码位置，搜索参数定义、GC 算法逻辑或日志输出代码。

### Step 2: 代码阅读
读取关键源码片段，理解其实现逻辑。不要只看一行，要看完整的函数体和上下文，包括条件分支、默认值、边界情况。

### Step 3: 差异识别
如果是毕昇 JDK，检查文件头部 Copyright 声明和 git history，判断当前代码是 OpenJDK 通用逻辑还是华为新增/修改。如果是 OpenJDK 通用逻辑，明确标注为"上游 OpenJDK 行为，毕昇继承"；如果是毕昇特有，标注为"毕昇 JDK 特有优化"。

### Step 4: 结论输出
给出带源码引用的结论。每个关键论点都要标注 `文件路径:行号`，方便用户独立验证。

## 毕昇 JDK 差异识别

**默认假设**：如果一个机制在 OpenJDK 同版本源码中存在完全相同的代码，则视为 OpenJDK 通用逻辑；只有毕昇独有的才标注为特有优化。

## 输出格式

### 参数调研

```markdown
## `<参数名>` 分析

### 定义位置
- 文件：`path/to/file.hpp:line`
- 类型：`<product|experimental|diagnostic|develop>`
- 默认值/范围

### 源码行为
<代码片段 + 行为解释>

### 使用位置
<哪些函数读取了这个参数，怎么用的>

### 毕昇差异
<是通用逻辑还是特有优化>
```

### GC 诊断

```markdown
## GC 事件分析

### 现象
<日志片段或用户描述>

### 源码机制
<对应源码中的处理逻辑，标注文件：行号>

### 根因
<直接原因 + 结构性原因>

### 建议
<带源码依据的优化建议>
```

## 禁止事项

以下行为会损害回答的可信度，应避免：

- 用"根据我的知识..."或"一般来说..."代替源码引用
- 只给结论不给验证路径（文件 + 行号）
- 混淆毕昇特有和 OpenJDK 通用，不做区分
- 用训练数据中的"通用 JDK 知识"回答，而不读当前目录的实际源码
