# 自检清单

落稿后逐条对照。

## 主线（分析阶段就要立对，表达阶段才不翻车——首查）
- [ ] 输出第一段是"根本目的/为什么"，而不是机制名称或源码？
- [ ] 是否建立了推导链（根本目的→必然设计），让每个机制在被讲到前读者已知它为什么必须存在？
- [ ] 每个机制都讲了"为什么存在"，而非只讲"做什么"？

## 证据（分析阶段就要做足，立身之本）
- [ ] 每个关键结论都有源码证据（`文件:行号`）并绑定论点？
- [ ] 没有用"一般来说""根据我的知识"、网上文章**替代源码验证**？（预判和查到的资料只是线索，结论只认源码）
- [ ] 源码看全了吗？Step 1 清单上预期的机制/边界，都追到调用链、核查了全部分支和默认值吗？——只看局部就敢下结论是最大的翻车点。
- [ ] 讲"为什么这样设计"时，有没有只凭源码现状反推（事后合理化）？设计意图该查 JEP/commit 吃透，但交付时用消化后的话讲，且毕昇是否照实现仍以源码为准。

## 表达（表达阶段查）
- [ ] 用因果链串起了知识点？
- [ ] 用了类比/分层/对比降低门槛？
- [ ] 复杂流程有图或表？
- [ ] 说明了联动关系？
- [ ] 结尾有一句话总结？
- [ ] 毕昇特有逻辑已单独标注？

## 完整性维度（按场景查，别漏）

**讲单个参数/机制时，别漏：**
- [ ] 它解决的根本矛盾（靶心）
- [ ] 默认值的来由（为什么是这个值，非拍脑袋）
- [ ] 联动关系（改它影响谁）
- [ ] 实践判断（什么场景调、调大调小的后果）

**讲一次事件/现象诊断时，别漏：**
- [ ] 现象本身（日志/描述）
- [ ] 根因（直接 + 结构性）
- [ ] 联动影响（和哪些参数/机制相关）
- [ ] 带源码依据的建议（非经验公式）

**讲一个完整系统/流程时，别漏（这最易翻车）：**
- [ ] **起点**：数据/对象怎么产生、初始状态是什么（如"对象分配时染什么颜色"）
- [ ] **触发**：何时启动、谁触发、对应用是否阻塞（如"GC 怎么被触发"）
- [ ] **主体流程**：核心阶段，每阶段 Why/What/How 三层
- [ ] **衔接/循环**：周期间如何延续、残局如何处理（如"没处理完的怎么留到下一轮"）——常被漏
- [ ] **归位表**：机制 ↔ 它解决的根本问题（反推检验每个机制都讲透了"为什么"）
- [ ] 有没有读者"不知道自己不知道"的盲区没填？（主动补，不只答字面提问）
